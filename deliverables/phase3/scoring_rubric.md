# Scoring Rubric: Backlog Synthesizer

## How Scoring Works

Each golden scenario is scored by running the system output through automated comparisons and LLM-as-judge evaluation. This rubric defines exactly how each comparison is performed.

---

## R1: Story Extraction Scoring (M1)

**Method:** Match system stories to golden stories by semantic similarity.

```
For each golden story:
  1. Find the best-matching system story (by title + description cosine similarity)
  2. If similarity > 0.85 → matched (true positive)
  3. If similarity 0.60-0.85 → partial match (review manually)
  4. If similarity < 0.60 → not found (false negative)

For each system story not matched to any golden story:
  → extra story (false positive — not necessarily wrong, but not in golden set)
```

**Score:** `matched / total golden stories`

**Manual review needed for:** Partial matches and extra stories (system may find valid requirements not in golden set).

---

## R2: Story Quality Rubric (M2, LLM-as-Judge)

Each story scored 1-5 on four dimensions:

### Clarity (1-5)
| Score | Description |
|---|---|
| 5 | Title and description are precise, unambiguous, single-purpose |
| 4 | Clear and understandable, minor rewording would improve |
| 3 | Generally clear but some ambiguous language |
| 2 | Vague, could be interpreted multiple ways |
| 1 | Unclear, would need significant rewriting |

### Actionability (1-5)
| Score | Description |
|---|---|
| 5 | Developer could implement immediately without questions |
| 4 | Mostly implementable, 1-2 minor clarifications needed |
| 3 | Implementable with some assumptions |
| 2 | Too vague to implement without significant clarification |
| 1 | Not actionable — more of a wish than a requirement |

### Completeness (1-5)
| Score | Description |
|---|---|
| 5 | Captures 100% of the requirement from source text |
| 4 | Captures 80%+, minor details missing |
| 3 | Captures the core but misses important details |
| 2 | Captures only part of the requirement |
| 1 | Misses the main point of the requirement |

### Acceptance Criteria Quality (1-5)
| Score | Description |
|---|---|
| 5 | All criteria are specific, testable, and sufficient for the story |
| 4 | Most criteria are testable, 1 could be more specific |
| 3 | Criteria exist but some are vague or untestable |
| 2 | Criteria are too generic to be useful |
| 1 | Missing or completely irrelevant criteria |

**LLM-as-Judge prompt:**
```
You are evaluating a story generated by an AI system from a meeting transcript.

Source text from transcript:
"{source_citation}"

Generated story:
Title: {title}
Type: {type}
Description: {description}
Acceptance Criteria: {acceptance_criteria}
Feature Tags: {feature_tags}

Score each dimension 1-5 using these rubrics:
[rubric definitions above]

Output JSON:
{
  "clarity": {"score": N, "rationale": "..."},
  "actionability": {"score": N, "rationale": "..."},
  "completeness": {"score": N, "rationale": "..."},
  "acceptance_criteria": {"score": N, "rationale": "..."},
  "overall": N.N
}
```

## R3: Feature Tag Scoring (M3)

**Method:** Set comparison between system tags and golden tags per story.

```
For each matched story pair (system vs golden):
  precision = |system_tags ∩ golden_tags| / |system_tags|
  recall    = |system_tags ∩ golden_tags| / |golden_tags|
  f1        = 2 * precision * recall / (precision + recall)

Tags are compared case-insensitive with fuzzy matching:
  "risk-engine" matches "risk_engine" matches "Risk Engine"
```

**Aggregate:** Average F1 across all matched stories.

---

## R4: Epic Assignment Scoring (M4)

**Method:** Exact match on epic assignment.

```
For each matched story pair:
  If system.epic_assignment == golden.epic_assignment → correct
  If system proposes new epic and golden says "proposed: X" → correct if names are semantically similar
  If system assigns existing epic but golden says proposed (or vice versa) → incorrect
```

**Score:** `correct / total matched stories`

---

## R5: Check Detection Scoring (M5, M6)

**Method:** Match system checks to golden checks by (story_title + check_type).

```
For each golden check:
  Find system check with same story (by title match) AND same check_type
  If found → true positive
  If not found → false negative (missed check)

For each system check not matched to any golden check:
  → false positive (spurious check)

Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
F1        = 2 * P * R / (P + R)
```

**M5:** Overall check precision and recall (all check types)
**M6:** F1 specifically for conflict-type checks (overlap, prior_decision, architecture)

---

## R6: Grounding Scoring (M7)

**Method:** Automated citation verification.

```
For each system story:
  1. Extract source_citation text
  2. Search for it in the original transcript
  3. Compute string similarity (Levenshtein ratio)
  
  If similarity ≥ 0.90 → valid citation
  If similarity 0.70-0.89 → paraphrase (acceptable)
  If similarity < 0.70 → not found (grounding failure)
```

**Score:** `(valid + paraphrase) / total stories`

---

## R7: Confidence Calibration Scoring (M8)

**Method:** Statistical comparison across confidence levels.

```
Group all stories by confidence level (high / medium / low)
For each group, compute:
  - Story extraction accuracy (matched to golden)
  - Tag F1
  - Epic accuracy

Check monotonic: high_accuracy > medium_accuracy > low_accuracy
```

**Score:** Pass if monotonic for at least 2 of 3 sub-metrics.

---

## R8: Backlog Hygiene Scoring (M9)

**Method:** Match system hygiene flags to golden flags by external_item_id.

```
For each golden hygiene flag:
  If system flagged same item → true positive
  If system missed it → false negative

For each system flag not in golden:
  → false positive

Precision = TP / (TP + FP)
```

---

## R9: Meeting Quality Scoring (M10)

**Method:** Direct comparison of numeric and categorical values.

```
ambiguity_delta = |system.ambiguous_count - golden.ambiguous_count|
actionability_match = system.actionability_score == golden.actionability_score

Score:
  ambiguity_delta ≤ 1 AND actionability_match → pass
  ambiguity_delta ≤ 2 OR actionability_match → acceptable
  else → fail
```

## Manual Validation: Scenarios 1 and 3

To validate this rubric works, manually score golden scenarios 1 (clean) and 3 (ambiguous):

### Scenario 1 (Meeting 1 — clean kickoff)
- **Expected:** 8 stories, 0 checks, high confidence, high actionability
- **Manual check:** Are the 8 golden stories reasonable extractions from the transcript? Are the expected tags correct? Is "0 checks" realistic for a clean meeting?
- **Result:** Golden dataset reviewed — stories match transcript content, tags align with ERIS components, no conflicts expected from a kickoff meeting. Rubric applies cleanly.

### Scenario 3 (Meeting 5 — ambiguous, contradicts M1)
- **Expected:** 8 stories, 7 checks (2 prior_decision, 3 ambiguity, 1 architecture, 1 overlap), low actionability
- **Manual check:** Are the 7 checks realistic? Are prior_decision conflicts correctly identified? Do ambiguity flags point to genuinely ambiguous text?
- **Result:** Golden dataset reviewed — meeting 5 has explicit contradictions ("5 roles" vs "3 roles") and references to kickoff decisions. 7 checks is a reasonable count. Low actionability matches the meeting's unresolved nature. Rubric handles ambiguous scenarios well.

Key decisions:

- Fuzzy tag comparison (case-insensitive, separator-agnostic)
- Citation verification uses Levenshtein ratio, accepts paraphrases at 0.70+
- LLM-as-judge with 4-dimension rubric and structured JSON output
